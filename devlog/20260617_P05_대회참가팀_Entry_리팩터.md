# 20260617_P05 — 대회 참가팀(Entry) 리팩터

> 배경: SaaS 전환 중 "Opponent 를 클럽별로 두는 게 맞나?"에서 출발. 대회는 공유이고
> 참가팀도 공유이므로, **상대팀을 클럽별 복제 대신 대회/부문의 참가팀(Entry)로 모델링**한다.
> 선행: [P04](20260617_P04_SaaS_PhaseB_라우팅_스코핑.md) B1 완료. 이 리팩터 후 B2(스코핑) 진행.

## 목표

- `Opponent` 를 **공유(global)** 로 — 클럽마다 양재/서초크루를 중복 생성하지 않는다.
- 한 대회/부문의 **참가팀 목록을 `CompetitionEntry`** 로(우리 팀이면 `team`, 외부팀이면
  `opponent`, 둘 다 없으면 이름만 dummy 도 가능). → 상대팀 = 같은 부문의 다른 entry 에서 도출.

## 모델 변경 (범위 한정)

| 모델 | 변경 |
|---|---|
| `Opponent` | **Phase A `club` FK 제거**, `name` 전역 unique 복원. (공유 외부팀 디렉터리) |
| `OpponentMatch` | **Phase A `club` FK 제거**. (공유 대회의 사실 — 클럽 무관) |
| `CompetitionEntry` | `team` nullable 로 + `opponent` FK(nullable) 추가 + `external_name`(이름만 dummy). **정확히 하나**(team xor opponent xor external_name) 제약. unique 재정의. |

- `Match.opponent`·`OpponentMatch.home/away` 는 **그대로 `Opponent` 참조**(공유가 됐을 뿐).
  Match 를 entry 참조로 바꾸는 전면 개편은 **비범위**(후속) — 표준 순위/경기 로직 보존.

## 스코핑 관점 (B2 대비)

- 공유(클럽 무관): `Competition`·`Division`·**`Opponent`**·**`OpponentMatch`**·
  opponent/dummy 타입 `CompetitionEntry`.
- 클럽 소유: `Team`·`Player`·`Match`·`Notice`·`GalleryItem`·`Award`·team 타입 `CompetitionEntry`(team→club).

→ 결과적으로 B2 에서 **Opponent/OpponentMatch 는 스코프 대상에서 빠진다**(이중작업 방지).

## 데이터 마이그레이션

1. `Opponent.club` / `OpponentMatch.club` 컬럼 제거(Phase A backfill 무효화). `Opponent.name` 전역 unique.
2. `CompetitionEntry`: `team` nullable 화 + `opponent`·`external_name` 추가.
3. **참가팀 backfill**: 기존 `Match`(our_team·opponent)·`OpponentMatch`(home·away)를 훑어
   각 (competition, division/age_group) 의 참가팀 entry 를 생성(우리 팀 entry + 상대팀 entry).
   division 이 없으면 null 로 둠(추후 정리).

## 단계

- **R1**: 모델 변경 + 마이그레이션 + backfill (이 문서 범위 핵심).
- **R2**: admin/폼/`load_seocho_knockout`·`seed` 갱신(상대팀 entry 동시 생성), 참가팀 목록 표시(선택).
- **R3**: 테스트(`matches/tests`·`competitions/tests`) 갱신, 검증, 커밋.

## 비범위 (후속)

- `Match` 를 entry 쌍 참조로 통합(우리/상대 모두 entry), `OpponentMatch` 를 Match 로 흡수.
- 참가팀 기반 순위표 재작성(현재는 Match/OpponentMatch 집계 유지).
- 외부팀(dummy)→실제 플랫폼 클럽 연결.
