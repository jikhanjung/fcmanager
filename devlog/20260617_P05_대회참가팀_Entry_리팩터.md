# 20260617_P05 — 대회 참가팀(Entry) 리팩터 **전면 개편**

> 결정(2026-06-17): 최소 범위가 아니라 **Match 를 참가팀(Entry) 쌍 참조로 전면 개편**.
> Opponent 공유화 + OpponentMatch 를 Match 로 흡수. 선행 B1 완료. 이후 B2 스코핑.

## 핵심 모델

### 참가팀 = `CompetitionEntry` (개념상 "참가팀")
한 대회/부문의 참가 단위. 우리 팀이거나 외부팀.
- `competition`, `division`(nullable)
- **`team`(→Team, nullable) XOR `opponent`(→Opponent, nullable)** — 정확히 하나.
- `__str__`/`name` = team.name 또는 opponent.name. `club` = team.club 또는 None.
- unique: `(competition, division, team)` / `(competition, division, opponent)`.

### `Match` = 두 entry 간 경기
- 기존 `our_team`+`opponent` → **`home_entry`·`away_entry`(→CompetitionEntry)**.
- 기존 `our_score`/`opponent_score` → **`home_score`·`away_score`**.
- **`OpponentMatch` 흡수**: 우리 팀 entry 가 없는 경기(상대팀 간)도 그냥 Match.
- `club`(기록 주체) 유지, `kickoff` **nullable 화**(시간 없는 보정 경기 허용).
- 피더: `opponent_feeder`·`advance_feeder` 둘 다 **self FK(Match)** 로 통일.

### 백워드 호환 프로퍼티 (뷰·템플릿·서비스 churn 최소화 ★)
Match 에 다음을 **property** 로 제공 → 기존 코드 거의 그대로:
- `our_entry`: home/away 중 `team.club == self.club` 인 entry(없으면 None — '상대팀 간 경기').
- `our_team` = our_entry.team · `opponent` = 반대편 entry · `is_home` = our_entry==home_entry.
- `our_score`/`opponent_score` = our_entry 기준 점수 · `result` = our 기준 W/D/L.
- `is_our_match` = our_entry is not None.

### 그대로 두는 것
- `MatchEvent.side`(OUR/OPPONENT, our_entry 기준) — 이벤트는 우리 경기에만 존재.
- `MatchLineup`(our_entry 팀). 중계 콘솔·라인업·스코어 입력은 our_team 프로퍼티로 동작.
- `Opponent`(공유로): **`club` FK 제거**, `name` 전역 unique. (외부팀 디렉터리)

## 데이터 마이그레이션

1. `Opponent.club` 제거 + 전역 unique. `CompetitionEntry` 필드 추가(team nullable, opponent).
2. Match 신규 필드(home/away entry·score) 추가, kickoff nullable.
3. **backfill**:
   - 각 Match: our_team·opponent 로 Entry 2건 get_or_create → is_home 에 따라 home/away 배정·점수 이전.
   - 각 OpponentMatch: age_group→division 매핑 후 Entry 2건 → Match 신규 생성(club·stage·score 이전,
     kickoff 없으면 null). 피더 관계 재연결(OpponentMatch→새 Match).
4. 구 필드(`our_team`·`opponent`·`our_score`·`opponent_score`) 및 `OpponentMatch` 모델 제거.

## 단계 (R1~R5)

- **R1**: CompetitionEntry/Opponent/Match 모델 변경 + 마이그레이션 + backfill + 호환 프로퍼티. `check`·스모크.
- **R2**: 서비스(`matches/services.py`)·순위/통계 집계를 home/away entry 기반으로 재작성.
- **R3**: 뷰·폼·템플릿 점검(호환 프로퍼티로 대부분 무변경, 상대팀 선택 UI 를 Entry/Opponent 로).
- **R4**: admin·`seed`·`load_seocho_knockout` 갱신(참가팀 entry 생성), `OpponentMatch` 잔재 제거.
- **R5**: 테스트 갱신, 전수 검증, 커밋. → 이후 SaaS B2.

## 위험

- 가장 큰 리팩터. **호환 프로퍼티**가 churn 방어선 — 깨지면 광범위 수정.
- 순위표/녹아웃 대진표/피더가 OpponentMatch 에 강결합 → R2/R3 에서 집중 검증.
- 운영 데이터는 단일 클럽(fcsky)뿐이라 backfill 검증은 단순하나, 마이그레이션은 비가역 — 사전 스냅샷.
