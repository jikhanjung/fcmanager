# gosu 권한 드롭 도입 — 디렉터리 마운트 소유권 함정의 근본 해소 (0.6.18)

- 날짜: 2026-07-14
- 선행: devlog 088(디렉터리 마운트 전환 + dolfinid 소유권 함정 → chown 대증 처치 + 쓰기
  프로브). 본 작업 = cdGTS entrypoint 동형의 **근본 해소**.

## 문제 재정리

- Dockerfile `USER appuser`(uid 1000 고정)는 호스트별 uid 매핑이 다르면 깨진다 —
  m710q 는 uid 1000=jikhanjung(자연 통과), dolfinid 는 uid 1000=ubuntu ≠ 배포계정
  honestjung(1001) → `db/` 소유가 어긋나면 `-journal` 생성 실패(쓰기 readonly).
  chown 은 대증 — 다음 이행·복원·신규 호스트에서 재발 여지.

## 해법 (cdGTS 동형)

- **Dockerfile**: `apt-get install gosu`(정적 ~1.2MB, `gosu nobody true` 설치 검증),
  `USER appuser` 제거(useradd 불요 — gosu 는 숫자 uid 로 동작).
- **docker-entrypoint.sh**: root 로 시작 → `/app/hostdb` 마운트 **소유 uid/gid 런타임 감지**
  → 이전 실행이 남긴 DB 형제 파일 소유 멱등 정리(chown) → `exec gosu <uid:gid> "$0" "$@"`
  재진입 → (비-root 로) migrate·superuser 보증 → `exec gunicorn`.
  - root 는 시작 순간에만 존재, 서비스 프로세스는 비-root — "root 웹서버" 리스크 없음.
  - su/sudo 대신 gosu 인 이유: TTY·fork 없이 그 자리에서 exec — 컨테이너 PID 1 에 적합.
  - 마운트가 root 소유면 root 폴백, 마운트 없으면(로컬 개발) 종전대로.
  - fcmanager 는 collectstatic 이 빌드 타임이라 cdGTS 와 달리 entrypoint 에서 불요.
- 효과: **"db/ 는 uid 1000 소유" 요건 소멸** — 임의 비-root 소유(1000·1001·기타)면
  한 이미지로 모든 호스트 커버, chown 수동 개입 불요. 0.6.17 쓰기 프로브는 안전망으로 존치.

## 검증

  `drop -> uid 1000:1000` 로그, DB 게이트+쓰기 프로브 OK, smoke PASS.
