# 20260715_094 — 백업 레인: 무결성 게이트 이식 + daily 미러의 torn copy 제거

> 성격: **운영 안전망.** cdGTS [devlog 150](../../cdGTS/devlog/20260715_150_hourly-integrity-gate.md)
> (0.1.68 파일럿)의 fcmanager 포팅 + 이식 과정에서 드러난 fcmanager 고유 결함 3건.
> 계기: 사용자 — *"cdgts 에서 오늘 있었던 문제 확인해서 이 프로젝트에 반영할 내용 있으면 반영해줘."*
> 릴리스: **0.6.24**

## 1. 왜 이게 "모니터링 추가"가 아닌가

`backup_db.py` 는 `src.backup(dst)`(online backup API)로 스냅샷을 뜬다. 이건 소스 페이지를 **충실히**
복사한다 — 검증하지 않는다. 그래서 소스가 깨져 있으면 스냅샷도 깨진 채 채택되고, `RETAIN_COUNT=12`
(매시)라 **12시간이면 성한 스냅샷이 전부 prune 된다.** 백업이 스스로를 오염시켜 **복구 대상 자체가
사라진다.** 손상을 늦게 아는 것보다 이쪽이 위험하다.

> 계약 §핵심원칙 3 이 운영 데이터의 안전을 통째로 백업에 위임하는 이상, 성립해야 하는 명제는
> "백업이 돌고 있다"가 아니라 **"백업이 복원 가능하다"** 다.

규칙은 두 줄(cdGTS 와 동일):

```
1. 스냅샷이 integrity_check 에 걸리면 → 채택하지 않는다 + prune 을 건너뛴다
2. db/INTEGRITY_FAIL 센티넬 → /healthz degraded(200) → smoke 실패
```

**(1) 이 본체다** — 아무도 안 보고 있어도 성한 스냅샷을 지킨다. **(2) 는 사람에게 닿는 경로**다.
dolfinid crontab 에 `MAILTO` 가 없어(cdGTS 와 동일 확인) cron 실패는 `backup/backup.log` 에만
남고 아무도 안 읽는다 — 읽히지 않는 검사는 연극이라, 사람이 이미 보는 `smoke` 에 물렸다.

## 2. 이식 전 실측한 것 (가정하지 않고)

cdGTS 판을 그대로 복사하면 안 되는 지점들을 먼저 확인했다.

| 확인 | 결과 |
|---|---|
| 센티넬을 cron 사용자가 쓸 수 있나 | **가능.** `db/` 는 `ubuntu:ubuntu drwxrwxr-x`, cron 은 honestjung — `id` 에 `1000(ubuntu)` 포함. 실제 `touch` 로 확인 |
| prod crontab `MAILTO` | **없음** — cdGTS 와 동일 전제 |
| 컨테이너 시야 | `/srv/fcmanager/db` → `/app/hostdb` 뿐(0.6.16). 센티넬이 `db/` 여야 하는 이유 동일 |
| 단일-서비스 가정 잔재 | `deploy.sh` FATAL 안내문의 `up -d --force-recreate web` 1건 — 회수 |
| prod 파일 교체 경로 | `deploy.sh`·`rollback.sh` 모두 서비스명 없는 `down` 후 교체 → 안전. `backup_db.py` 는 정지 불요 API |

## 3. tar 트랙에도 채택 게이트가 필요하다 (fsis 와 독립 수렴)

cdGTS 는 소스가 DB 하나지만 fcmanager 는 **nginx conf tar 트랙**이 있다(150 §11 이 "조사 선행 필요"로
남긴 부분). 같은 원칙을 적용하되 대응물을 골랐다 — **fsis2026 도 같은 날 같은 결론에 독립 도달**
(0.5.85: *"같은 MUST 가 같은 모양의 버그를 덮는다 — 채택 전 검증은 sqlite 전용 규칙이 아니다"*).
계약 §백업 레인에 이미 승격돼 있다:

| 트랙 | 채택 전 검증 | 센티넬 |
|---|---|---|
| DB | `PRAGMA integrity_check` | 올린다 (앱 상태 = degraded) |
| nginx tar | **`tar -tzf`** (gzip CRC + 구조) | 안 올린다 — conf 아카이브 손상은 서빙 상태와 무관 |

두 트랙 모두 **새 아티팩트를 못 만들었으면 prune 하지 않는다.** 종전 코드는 `backup_one()`/
`backup_data()` 의 반환값을 버리고 무조건 prune 했다 — 계약이 "5개 프로젝트 전부 이 모양"이라고
지적한 그 코드였고, fcmanager 는 그게 **두 트랙에** 있었다.

## 4. 진짜 수확은 daily 미러였다 — 가장 값진 백업이 가장 검증이 없었다

무결성 게이트를 이식하다 `backup-fcmanager.sh`(m710q daily 05시)를 읽고 발견했다:

```bash
scp "${REMOTE}:.../db/db.sqlite3"      "${DB_SNAPSHOT}"        # ← 가동 중인 라이브 WAL DB
scp "${REMOTE}:.../db/db.sqlite3-wal"  "${DB_SNAPSHOT}-wal"    # ← 그 다음에 따로
scp "${REMOTE}:.../db/db.sqlite3-shm"  "${DB_SNAPSHOT}-shm"    # ← 또 그 다음에 따로
```

라이브 WAL DB 를 세 번에 나눠 복사한다. 사이에 체크포인트가 끼면 본체와 -wal 이 어긋난 **torn
스냅샷**이 된다. 그리고 이 트랙이 먹이는 곳이:

- `~/backups/fcmanager/db_history/` — **30일 계층 보관**
- `/nas/.../fcmanager_backup/` — **90일 오프사이트**
- `~/dev_data/fcmanager/` — 테스트 미러

즉 **가장 오래 보관하는 백업이 일관성 보장도 검증도 없었다.** hourly 는 online backup API 를 쓰는데
daily 는 `cp` 하고 있었던 것 — cdGTS `sync-cdgts-db.sh` 가 prod 에서 `src.backup()` 을 돌려 pull 하는
것과 대조적이다.

**고침**: 라이브 DB 대신 **운영 hourly 스냅샷을 pull** 한다. 오늘 만든 게이트 덕에 그건 이미
*integrity 를 통과한 일관된 단일 파일*이다. 받은 뒤 **로컬에서 한 번 더** `integrity_check` 하고,
통과할 때만 채택하며, 실패하면 `cleanup_tiered` 에 도달하지 않는다(= 30일/90일 창을 안 깎는다).
대가는 최대 1시간 staleness — daily 트랙에서 torn 과 바꿀 값이 아니다.

### 부수 효과: 배포와 무관한 두 번째 인간 채널

cdGTS 150 §11 이 부채로 남긴 것 — *"센티넬은 **배포 시점**에만 사람에게 닿는다. 배포가 뜸하면 탐지도
뜸하다."* fcmanager 는 여기서 닫힌다. daily 가 **스냅샷 신선도**를 본다:

```
최신 hourly 스냅샷이 2시간 넘게 낡음
  = hourly cron 중단  또는  무결성 게이트가 손상 스냅샷의 채택을 막는 중(= 손상 신호)
  → ERROR → telegram (m710q ~/scripts/notify-telegram.sh, 이미 있던 채널)
```

배포를 안 해도 **매일** 사람에게 닿는다. 센티넬(배포 때) + 신선도(매일) 두 경로.

## 5. dev_data 은퇴 — 그리고 149 가 기다리던 자리

사용자 지시로 daily step 8 의 목적지를 `~/dev_data/fcmanager` → **m710q 테스트 타깃
`/srv/fcmanager/db`** 로 바꿨다(미러가 둘이면 드리프트만 는다. `run-testserver.sh` 런처는 은퇴,
테스트는 `:8005` 도커 타깃으로 일원화).

그런데 **그 디렉터리는 테스트 컨테이너가 WAL 로 쥐고 있다**(`docker ps`: Up 27 hours). 그대로
복사했으면 [devlog 149](../../cdGTS/devlog/20260715_149_order-node-and-clamp-category-removal.md)
addendum 의 손상이 **fcmanager 에서 그대로 재현**됐을 것이다(cdGTS 는 그걸로 테스트 DB 를 두 번 깼다).

```bash
docker compose down          # 서비스명 없이 = 전 서비스. 사이드카가 생겨도 안 빠진다
cp "${DB_SNAPSHOT}" db/db.sqlite3
rm -f db/db.sqlite3-wal db/db.sqlite3-shm   # 정지 전 잔재를 남기면 새 본체에 낡은 wal 이 얹힌다
docker compose up -d
```

그리고 **정지에 실패하면 교체하지 않는다**(`WARN` 후 skip). 라이브 교체 금지가 규율이 아니라
제어 흐름이어야 한다.

## 6. 게이트가 또 자기 자신을 물었다 (cdGTS §10 의 재방문)

단위 테스트 20종이 전부 통과한 뒤 **실제로 돌려보니** 부산물이 나왔다:

```
db_20260715.sqlite3.tmp-shm    ← 32KB
db_20260715.sqlite3.tmp-wal    ← 0B
```

원인은 cdGTS §10 과 **완전히 같다** — 내가 방금 넣은 `integrity_check` 의 `mode=ro` 커넥션이 WAL
스냅샷의 `-shm` 을 만들어놓고 치울 권한이 없었다. (운영 hourly 스냅샷이 아직 **구 코드가 만든 WAL
모드**라 조건이 살아 있었다. 0.6.24 배포 후엔 소멸하지만, 그때까지 매일 2개씩 샜을 것이다.)
게다가 이름이 `*.tmp-shm` 이라 `cleanup_tiered` 의 glob(`db_*.sqlite3`)에도 안 걸려 **영구 누적**.

→ 해법: 검사 커넥션을 **rw 로 연다.** 대상은 라이브 DB 가 아니라 방금 받은 우리 임시 파일이므로
mode=ro 를 고집할 이유가 없다. 마지막 커넥션이 정상 종료하면 sqlite 가 -wal/-shm 을 스스로 지우고,
덤으로 `journal_mode=DELETE` 로 내려 **아카이브가 운영 코드 버전과 무관하게 항상 단일 파일**이 된다.

> **교훈이 문서로 적혀 있어도 새 코드는 같은 함정을 다른 자리에서 다시 밟는다.** 150 §10 을 읽고
> `backup_db.py` 엔 `journal_mode=DELETE` 를 넣었으면서, **daily 스크립트에 넣은 검사**는 같은 함정에
> 걸렸다. 잡은 건 테스트가 아니라 **한 번 돌려본 것**이다(`ls db_history/`).

## 7. 테스트 — 통과가 아니라 "무엇을 잡는가"

Django `manage.py test` 로 함께 돈다(pytest 미도입 프로젝트 — cdGTS 와 다른 점). **77 → 102 (+25)**.

- `scripts/test_backup_db.py` 20종 — 로테이션 오염/센티넬/tar 게이트/prune 보존
- `config/test_healthz.py` 5종 — ok·degraded(200)·unhealthy 우선순위·센티넬 규약 일치

통과만으론 의미가 없어 **게이트를 하나씩 빼보고(변이) 해당 테스트가 실패하는지** 확인했다:

| 변이 | 결과 |
|---|---|
| `journal_mode=DELETE` 제거 | ✔ `test_backup_leaves_no_wal_siblings` 실패 — `.tmp-shm`/`.tmp-wal` 검출 |
| 무조건 prune(종전 코드 모양)으로 되돌림 | ✔ `TestMain` 2종 실패 |
| 채택 게이트 제거 | ✔ `test_corrupt_source_is_not_promoted` 실패 |

첫 변이가 중요하다: cdGTS 는 이 조건을 **단위 테스트로 못 잡았다**(합성 DB 가 기본 저널 모드라
재현 조건이 픽스처에 없었다). 픽스처를 **명시적으로 WAL 로** 올려 단위 테스트 안으로 끌어왔다.
단언도 화이트리스트(`*.tmp` 없음)가 아니라 **전수**(`iterdir()` 에 dest 외 아무것도 없음)로 —
화이트리스트는 내가 상상한 실패만 잡는다.

## 9. 부채 / 안 한 것

- **`rollback.sh --db=restore` 는 복원할 스냅샷을 검사하지 않는다** — 오염된 로테이션에서 골라
  복원하면 손상을 되살린다. 이번 게이트가 오염 자체를 막아 위험은 크게 줄었지만 복원 직전
  `integrity_check` 1회는 값싼 방어다. 계약 §백업 레인이 "아직 어느 프로젝트도 구현하지 않았다"고
  적은 미해결 항목 — 3-repo 공통으로 다음 라운드.
- **센티넬 stale 창**: 복구 후 다음 정시까지 최대 1시간 degraded(smoke 계속 실패). 자동 해제를
  rollback 에 넣지 않은 건 의도 — 복원한 스냅샷이 성한지 검증 없이 플래그만 지우면 거짓 `ok` 다.
  급하면 `rm /srv/fcmanager/db/INTEGRITY_FAIL`(smoke 안내문에 명시).
- **`backup-fcmanager.sh` 는 self-heal 이 없다** — 실행본은 `~/scripts/`, 정본은 repo. 이번에 실제로
  **드리프트가 확인**됐다(repo 사본만 0.6.16 구 DB 경로에 머물러 있었고, telegram notify·NAS timeout
  도 호스트에만 있었다). 이번엔 repo ← 호스트로 흡수하고 헤더에 경고를 박았지만, 근본 해소(이미지
  self-heal 레인에 편입)는 안 했다. m710q 는 이미지를 소비하는 호스트가 아니라서 구조가 다르다.
- ~~**fsis2026 미적용** — 3-repo 정렬 잔여 1건.~~ **정정(2026-07-15 저녁, 계약 문서 반영 중 발견)**:
  fsis2026 은 이 작업과 **거의 동시에**(19:40) 이식을 마쳤다(0.5.85, fsis devlog 220) — 즉
  cdGTS·fsis·fcmanager **3개가 같은 날 각자 이식**했고, 셋이 서로를 모른 채 각각 새 항목을 보탰다.
