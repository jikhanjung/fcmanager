# 20260617_P03 — 단일 클럽 → 멀티테넌트 SaaS 전환 (최소 계획)

## 목표

지금은 FC Sky **한 클럽 전용**으로 만들어진 사이트를, 여러 축구 클럽이 각자
선수단·대회·일정·결과·중계를 운영하는 **멀티테넌트 SaaS**로 확장한다.
이 문서는 **최소 전환**(MVP)의 범위·전략·단계만 정의한다. 결제·커스텀 도메인 등은 비범위.

## 전략 결정 (최소·저위험)

| 항목 | 결정 | 이유 |
|---|---|---|
| 테넌시 격리 | **공유 DB + 행 단위(Club FK)** | SQLite 유지, 인프라 변경 0. schema/DB-per-tenant 는 과함 |
| 테넌트 식별 | **경로 기반 `/<club-slug>/...`** | 기존 `DJANGO_URL_PREFIX`(서브패스) 메커니즘 재사용 |
| 권한 | **`ClubMembership(user, club, role)`** | 전역 `is_staff` 대신 클럽별 운영진. superuser=플랫폼 관리자 |
| 스코핑 | **미들웨어 `request.club` + 매니저/믹스인 강제** | 쿼리 누락(데이터 누출) 방지를 한 곳에서 |
| 브랜딩 | **Club(name·slug·logo) + context processor** | 하드코딩 'FC Sky' 제거 |

## 현재 "단일 클럽" 가정이 박힌 곳

- **루트 모델에 클럽 소속 개념 없음**: `Team`, `Player`, `Competition`, `Opponent`,
  `Match`, `Notice`, `GalleryItem`. (하위 모델 `TeamMembership`·`Division`·`Award`·
  `MatchEvent/Lineup/Video`·`OpponentMatch` 는 부모 FK 로 스코프 상속)
- **전역 권한**: Django `is_staff` 가 사이트 전체 운영진을 의미.
- **브랜딩 하드코딩**: `templates/base.html`·`home.html` 등 다수에 'FC Sky' 문자열.
- **URL**: 전체 사이트가 단일 `URL_PREFIX=FcSky` 하위.

## 데이터 모델 변경

- **신규 `Club`(tenant)**: `name`, `slug`(URL), `logo`, `is_active`, 생성일.
- **루트 모델에 `club` FK 추가**(나머지는 부모 통해 상속):
  `Team`, `Player`, `Competition`, `Opponent`, `Match`, `Notice`, `GalleryItem`.
  - 추가 절차: `null=True` 로 추가 → 기존 데이터 backfill → `null=False` 로 확정.
- **신규 `ClubMembership`**: `(user, club, role)` — role ∈ {owner, staff}. `unique_together`.
- **유니크 제약 재정의**: 전역 unique(예: `Team.slug`, `Opponent.name`, `Competition.slug`)를
  **`(club, slug)` 복합 unique** 로 변경(클럽 간 동일 이름 허용).

## 요청 처리 / 스코핑

- **미들웨어**: URL 첫 세그먼트(`/<club-slug>/`)로 `Club` 조회 → `request.club`. 없으면 404.
- **URL 구조**: 기존 앱 URL 을 `/<club-slug>/` 하위로 묶는다(현 `URL_PREFIX` 래핑 일반화).
  플랫폼 레벨 경로(`/`, `/signup/`, `/admin/`)는 테넌트 밖.
- **쿼리 스코핑**: 공개·관리 뷰 모두 `request.club` 으로 필터. 루트 모델에
  `objects.for_club(club)` 헬퍼(매니저) 제공, 뷰 믹스인으로 강제.
- **쓰기**: 생성 시 `club=request.club` 자동 주입(폼/뷰 베이스에서).

## 권한 / 온보딩 (최소)

- 클럽 운영진 판정: `is_staff` → **`request.user` 가 `request.club` 의 `ClubMembership(staff/owner)` 인지**로 교체.
- **클럽 생성(signup)**: 사용자가 클럽명·슬러그 입력 → `Club` + `ClubMembership(owner)` 생성.
- superuser 는 전 클럽 접근(플랫폼 운영). 셀프서비스 자동화·이메일 인증은 후속.

## 데이터 마이그레이션 (기존 FC Sky 보존)

1. `Club(slug="fcsky", name="FC Sky")` 1건 생성(데이터 마이그레이션).
2. 모든 루트 모델 기존 행 `club=fcsky` backfill.
3. 기존 운영진 user → `ClubMembership(club=fcsky, role=owner/staff)`.
4. 운영 배포: `DJANGO_URL_PREFIX=FcSky` → `/fcsky/` 슬러그로 매핑(또는 리다이렉트 유지).

## 단계 (Phase 분할)

- **A. 테넌트 모델 + backfill** — `Club`/`ClubMembership`, 루트 FK(nullable→backfill→확정),
  복합 unique. admin 등록. (DB·마이그레이션만, 동작 변화 없음)
- **B. 라우팅 + 스코핑** — 미들웨어 `request.club`, URL `/<slug>/` 하위화, 전 뷰 쿼리 필터·쓰기 주입.
- **C. 권한 + 온보딩** — 클럽별 staff 체크로 교체, 클럽 생성·멤버 초대 최소 화면.
- **D. 브랜딩 분리** — context processor 로 'FC Sky' → `request.club.name`, 로고/테마 per club.

각 Phase 끝에 FC Sky(=`fcsky` 클럽) 단독으로도 기존과 동일하게 동작하는지 회귀 확인.

## 비범위 (후속)

- 결제·요금제·쿼터, 커스텀 도메인/서브도메인, schema/DB-per-tenant 격리,
  PostgreSQL 전환(다중 클럽 트래픽 시), 셀프서비스 가입 자동화·이메일 인증,
  테넌트별 백업 분리.

## 리스크 / 주의

- **데이터 누출**: 스코핑 누락이 곧 타 클럽 데이터 노출. → 중앙 매니저/믹스인 + 테스트로 강제.
- **권한 경계**: 전역 `is_staff` 잔재가 클럽 경계를 무력화. → 전수 교체 + 테스트.
- **URL 변경 파급**: 기존 `/FcSky/` 링크·북마크·nginx·배포 설정. → 리다이렉트로 흡수.
- **SQLite 동시성**: 클럽·트래픽 증가 시 한계 → Postgres 전환(별도 과제).
- **Opponent 공유 여부**: 최소안은 클럽별 분리. 공용 상대팀 풀은 후속 검토.
