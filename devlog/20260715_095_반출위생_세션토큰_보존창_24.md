`auth_user` 는 **admin 1명**(제3자 PII 0) · 해시는 pbkdf2 — cdGTS 와 같은 지형이다. 문제는 `django_session`.

## 2. 왜 위험한가 — `SECRET_KEY` 가 달라도 안 막힌다

`session_key` 는 **쿠키에 담기는 값 그 자체(bearer 토큰)** 다.

```
사본에서 session_key 를 읽는다 → 쿠키에 담아 **운영**(fcmanager.app)에 제시
  → 운영이 자기 DB 에서 그 행을 찾고 자기 SECRET_KEY 로 디코드 → 로그인됨
```

*"운영과 테스트의 SECRET_KEY 가 다르니 안전"* 은 **틀렸다.** 키 차이가 막는 건 "받는 쪽이
`session_data` 를 **해독**하는 것"뿐인데, 이 공격은 해독을 하지 않는다 — 훔친 키를 **운영에**
돌려주면 운영이 자기 키로 알아서 푼다. **해시는 뚫어야 쓰지만 세션 키는 그냥 쓰면 된다.**

대칭적으로 그 세션들은 테스트 컨테이너에선 무용지물이다(서명 검증 실패 → 빈 세션).
= **테스트엔 쓸모없고 운영엔 위험한 순수 손해.** 지워서 잃는 게 없다.

## 3. 어디서 지우나 — cdGTS 와 다른 자리를 골랐다

cdGTS 는 `sync-cdgts-db.sh`(반출 스크립트)에 넣었다. **그쪽은 sync 가 자기 스냅샷을 직접 뜨기 때문**이다.
fcmanager 는 094 에서 daily 가 **hourly 스냅샷을 소비**하도록 바꿨으므로 **초크포인트가 상류로 옮겨왔다**:

```
backup_db.py (hourly, 운영)  ← 여기서 지우면
   ├→ daily 미러 → ~/backups(30일) → NAS(90일·0777)   ┐
   ├→ 테스트 컨테이너 /srv/fcmanager/db                 ├ 전부 구조적으로 깨끗
   └→ 누가 손으로 scp 해가도                            ┘
```

계약 MUST 는 *"반출 직전에"* 지우라고 하는데, fcmanager 에선 **hourly 스냅샷이 곧 반출물**이다
(그게 호스트를 떠나는 유일한 DB 사본). 그래서 여기가 "직전"이면서 동시에 **유일한 초크포인트**다.

**대가와 그게 괜찮은 이유**: hourly 로 복원하면 전원 재로그인. 그런데 **rollback 경로는
pre_deploy 스냅샷**(정지 후 `cp`, 호스트를 떠나지 않음)이라 **그쪽 세션은 온전하다**. hourly 는
재해 복구용이고, 재해 복구 시 재로그인은 무시할 만한 비용이다. ★ **라이브 DB 는 읽기만**
(백업 API 는 소스를 안 건드린다) — 실측: 위생 전후 운영 라이브 34행·유효 2개 **불변**.

**소비자 쪽에도 한 겹**(`backup-fcmanager.sh`): 운영이 이미 지우고 내보내지만 그건 **상대 코드
버전에 기댄 가정**이다(구버전 운영·수동 스냅샷). NAS 0777 로 밀어넣는 당사자가 자기 책임으로
한 번 더 확인한다 — 094 §6 의 *"생산자만 고치면 끝이 아니다"* 와 같은 논리. 이미 깨끗하면 0행 = 무해.

## 4. VACUUM — 통설을 실측이 정정했다

cdGTS 151 과 계약은 *"DELETE 만으론 free page 에 토큰 문자열이 남는다"* 고 적었고 나도 그대로 옮겼다.
**이 환경에선 틀렸다**:

| 조건 | 파일에 토큰 잔존? |
|---|---|
| DELETE 만 (기본 `secure_delete=1`) | **아니오** |
| DELETE + VACUUM | 아니오 (77KB → 12KB) |
| `secure_delete=OFF` + DELETE 만 | **예** ← 통설이 성립하는 조건 |
| 위 + VACUUM | 아니오 |

m710q·dolfinid **둘 다 `PRAGMA secure_delete` 컴파일 기본값이 1**(sqlite 3.45/3.46 실측)이라
DELETE 만으로도 바이트가 지워진다. **그래도 VACUUM 을 둔다** — 이유가 바뀔 뿐이다:
그건 **주변 환경의 기본값에 기댄 것이지 계약이 아니다**(`secure_delete=0` 빌드에선 통설대로 남는다).
VACUUM 은 그 기본값과 무관하게 free page 자체를 없앤다. **주석을 사실대로 고쳐 적었다** —
틀린 근거로 옳은 코드를 지키면 다음 사람이 그 근거를 믿는다.

## 5. 테스트가 두 번 공허했다

**"통과"는 신호가 아니다.** 변이 테스트(게이트를 빼고 실패하는지 확인)가 두 번 잡아냈다:

1. `test_token_bytes_are_gone_from_file` — VACUUM 을 빼도 **통과**했다(§4 의 `secure_delete=1` 때문).
   즉 내가 "VACUUM 을 검증한다"고 믿은 테스트는 아무것도 검증하지 않았다.
2. 그래서 `freelist_count == 0`(= 파일이 재작성됨) 단언을 추가했는데 — **이것도 통과**했다.
   세션 **3행짜리 픽스처는 한 페이지에 들어가** DELETE 해도 루트 페이지가 테이블에 남아
   freelist 가 0 이었다. 픽스처를 **200행**으로 키우자 비로소 잡혔다(`17 != 0`).

> 픽스처가 재현 조건을 안 만들면 단언은 장식이다. 094 §7 에서 cdGTS 의 WAL 픽스처 결함을
> 지적해놓고 **같은 라운드에서 두 번** 같은 실수를 했다.

## 6. 보존 창 12 → 24 — 튜닝이 아니라 유도값

```
RETAIN_COUNT × 주기 ≥ 오프사이트 간격
  12 × 1h = 12h  <  24h  ✗   ← 05시 daily 와 17시 사이는 시간 단위 복원 불가(어제치 daily 로만)
  24 × 1h = 24h  ≥  24h  ✓
```

세밀 트랙(hourly)의 창이 성긴 트랙(daily 05시)의 간격을 **덮어야** 두 트랙 사이에 granularity 갭이
없다. 5개 프로젝트가 전부 hourly+daily 라 **24 는 공통 유도값**이다. 비용 0.45MB × 24 ≈ **11MB**.

## 7. 소급 정리 — 새 사본만 고쳐선 안 닫힌다

**62벌 전부**에서 세션 1,992행 제거(DELETE + VACUUM). 테스트 타깃 1벌은 **컨테이너가 WAL 로 쥐고
있어** 094 §5 의 규칙대로 `down`(서비스명 없이) → 위생 → `-wal`/`-shm` 제거 → `up` 으로 처리.
사후 전수 재확인 **0행 0유효**, 데이터 무사(28경기·integrity ok), 운영 라이브 무영향.

`dev_data` 는 은퇴(094 §5)했는데 **파일이 남아 유효 토큰 2개를 들고 있었다** — 위생은 했지만
**디렉터리 자체를 지우는 게 맞다**(§9).

## 9. 부채 / 안 한 것

- ⚠️ **NAS 가 `drwxrwxrwx`(0777)** — 세션 토큰은 뺐지만(무엇을 내보내나) **어디에 두나는 그대로**다.
  강한 해시·운영 데이터가 여전히 world-readable 에 90일 산다. cdGTS 151 §9 와 **같은 열린 항목**이고
  NAS 공유 설정은 3-repo 공용이라 별건.
- **`~/dev_data/fcmanager` 디렉터리 제거** — 은퇴했고 위생도 했지만, 갱신되지 않는 미러가 남아 있으면
  다음 사람이 그걸 최신으로 착각한다. 삭제 필요(사용자 확인 후).
- **운영 `backup/` 의 구 스냅샷**(위생 전 것)은 손대지 않았다 — 운영 호스트 = 라이브 DB 와 같은 신뢰
  경계이고 24h 내 로테이션으로 빠진다. 그 사이 daily 가 pull 하는 건 **최신**(= 위생된 것)이다.
- `rollback.sh --db=restore` 스냅샷 미검증(094 §9) — 3-repo 공통, 여전히 미해결.
